  Wire JWT refresh endpoint and extract token validation middleware

  ## Summary
  Wired up the missing /api/refresh endpoint in src/routes/auth.ts so
  clients can swap a refresh token for a fresh 15-minute access token
  without re-logging in. JWT signing and verification are now shared
  from a new src/middleware/jwt.ts module instead of living inline in
  the login handler.

  ## What happened
  Started in src/routes/auth.ts, which previously only had
  loginHandler() signing an access token inline with a top-level
  `jsonwebtoken` import and a `JWT_SECRET || 'dev-secret'` constant.
  There was no refresh path at all — the API contract the user wanted
  (POST /api/refresh -> new access token) didn't exist yet.

  First instinct was to add the refresh handler in the same file and
  call `jwt.verify` inline, but that would duplicate the signing /
  verifying logic that loginHandler already owned. So the JWT work
  was extracted into src/middleware/jwt.ts as two small helpers:
    - jwtVerify(token): wraps jwt.verify in a try/catch and returns
      the decoded payload (`{ sub }`) on success or null on failure,
      so route handlers can branch on a single truthiness check.
    - jwtSign(payload): wraps jwt.sign with the 15m expiry so every
      caller gets the same TTL without repeating the options object.
  The SECRET constant moved into the middleware module and the
  `jsonwebtoken` import moved with it; routes/auth.ts now only
  imports { jwtVerify, jwtSign } from '../middleware/jwt'.

  The reason the middleware lives under src/middleware/ and not
  src/lib/ is a circular-import trap the user flagged: putting JWT
  helpers next to other lib modules would have pulled routes/auth.ts
  -> lib/db -> jwt back into the same dependency cycle the extraction
  was meant to break. Keeping jwt.ts on the middleware side of the
  graph sidesteps it entirely.

  With the middleware in place, refreshHandler became ~10 lines in
  src/routes/auth.ts: pull refreshToken off req.body, 400 if missing,
  jwtVerify it, 401 if invalid, otherwise jwtSign a new access token
  keyed on the same `sub` and return it as { accessToken }. README.md
  was added with a short API section listing both /api/login and the
  new /api/refresh so the contract is discoverable from the repo root.

  .claude/settings.json was also created this session — the repo had
  no Claude settings at all, so a SessionStart hook was added that
  dumps the last 5 commits on startup. That hook is what makes this
  commit body useful to tomorrow's fresh session.

  ## Files touched
  - src/routes/auth.ts: replaced the inline `jsonwebtoken` import and
    SECRET constant with `import { jwtVerify, jwtSign } from
    '../middleware/jwt'`. loginHandler now calls jwtSign instead of
    jwt.sign. Added refreshHandler: validates req.body.refreshToken,
    returns 400 on missing, 401 on invalid, otherwise issues a new
    access token via jwtSign({ sub: payload.sub }).
  - src/middleware/jwt.ts: new file. Holds the SECRET constant and
    the two shared helpers jwtVerify / jwtSign. Placed under
    src/middleware/ (not src/lib/) specifically to avoid the
    routes/auth -> lib/db -> jwt circular-import path.
