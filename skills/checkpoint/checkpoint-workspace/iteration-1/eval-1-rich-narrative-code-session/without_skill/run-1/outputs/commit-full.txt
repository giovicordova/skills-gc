Debugging note — do not repeat this mistake
-------------------------------------------
~15 minutes were lost chasing the sort order. sorted() was being called
on a generator returned by _load_from_db() that had already been
partially consumed by an earlier debug print. The fix was to wrap the
call in list() before sorting so the sequence is materialised exactly
once. If list_widgets() ever returns empty or wrong-ordered results
again, check first whether _load_from_db()'s generator is being
iterated somewhere upstream (logging, debug prints, assertions).

Rejected: functools.lru_cache on _load_from_db()
------------------------------------------------
User proposed adding @functools.lru_cache to _load_from_db() for
performance. We discussed and explicitly rejected it. Reason: widgets
are mutated in the database out-of-band (other processes write to the
same table), so a cached view would go stale and return incorrect data
— worse than the current uncached behaviour. Do NOT reintroduce
lru_cache here without first solving cache invalidation against
out-of-band writers. This decision is intentional, not an oversight.
