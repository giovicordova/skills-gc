    - Added `tests/test_filters.py` covering the new filter.
    
    The sort bug (dead end worth remembering)
    Initial implementation looked right but returned widgets in the wrong order. About 15 minutes were lost chasing the sort itself: added `reverse=True` to `sorted()`, then moved it to the inner call, then tried wrapping the result in `reversed()` — none of it changed the output. Root cause was nothing to do with sorting: `_load_from_db()` returns a generator, and a debug `print` statement a few lines above the sort had already consumed it, so `sorted()` was operating on an empty iterable and silently producing nothing useful. Fix was to materialise the generator with `list(_load_from_db())` before sorting. Lesson for next session: when a sort "doesn't work" on data from `_load_from_db()`, check for upstream iteration first — the function returns a generator, not a list.
    
    Rejected: lru_cache on _load_from_db()
    The user suggested wrapping `_load_from_db()` in `@functools.lru_cache` for performance. We discussed and decided against it. Widgets are mutated in the database out-of-band (other processes / manual edits), so a cached result would go stale and return wrong data — worse than the current uncached cost. This decision should not be revisited without first solving the out-of-band mutation problem. Recording it explicitly so a future session doesn't reintroduce caching as an "obvious" optimisation.
