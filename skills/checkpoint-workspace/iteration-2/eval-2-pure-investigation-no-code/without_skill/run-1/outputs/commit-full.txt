## Problem
Users on staging are logged out on every request. Prod is unaffected.

## Investigation trail
1. Checked app/auth.py — cookie is set on `.example.com` (parent domain),
   which should cover both staging and prod. Code is correct.
2. Inspected browser devtools behaviour (reported by user): the Set-Cookie
   header is returned on every response, but the browser does not send the
   cookie back on the next request. Chrome is silently rejecting it.
3. Traced the path through infra/nginx/staging.conf. The staging nginx
   proxies to the app without rewriting the Set-Cookie domain. The app
   emits `domain=.example.com`, but the browser is on
   `staging.example.com` and the upstream uses a different internal
   hostname. Chrome's cookie jar rejects the parent-domain cookie because
   the TLS SNI and the cookie domain do not cleanly align against
   staging's wildcard cert.

## Root cause
infra/nginx/staging.conf is missing a `proxy_cookie_domain` directive.
The app code is NOT the problem.

## Decision
Do NOT touch app/auth.py — it is correct.
Fix belongs in infra/nginx/staging.conf via `proxy_cookie_domain`.
Giovanni will apply the nginx change himself on Monday during the
maintenance window. Today's session ends with diagnosis only; zero files
were modified.
